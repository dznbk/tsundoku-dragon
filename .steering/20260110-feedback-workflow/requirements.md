# 実装後フィードバック対応ワークフロー

## 概要

`/add-feature` で機能を実装した後、ユーザーからのフィードバック（UX改善、バグ修正など）に対応する際のワークフローを定義するスキルを追加する。

## 背景

- `/add-feature` は新規ステアリング作成 → 実装 → PR の流れが確立されている
- 実装後のフィードバック対応をどこに記録すべきか不明確
  - 完了したステアリングに追記？
  - CONTEXT.md？
  - 新しいステアリング？
- 本の登録画面実装後のフィードバック対応（CORS修正、タイトル自動入力、スキルUX改善）で課題が発生

## 要件

### 機能要件

- [ ] フィードバック対応用の `/continue-feature` コマンドを新設
- [ ] 既存のステアリングを参照しながら、追加タスクを管理できる
- [ ] 軽微な修正か継続作業かを判断する基準を明文化

### 非機能要件

- 既存の `/add-feature` ワークフローとの一貫性を保つ
- ステアリングスキルの3モード（計画・実装・振り返り）を活用

## 受け入れ条件

- [ ] `/continue-feature` コマンドが作成されている
- [ ] コマンドが既存ステアリングを参照して追加タスクを管理できる
- [ ] 「軽微な修正」と「継続作業」の判断基準がドキュメント化されている
- [ ] ステアリングスキルにフィードバック対応の運用ルールが追記されている

## 対象外（スコープ外）

- `/fix-bug` など別種のワークフロー（今回はフィードバック対応のみ）
- 既存ステアリングファイルの構造変更
