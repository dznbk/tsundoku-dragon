# タスクリスト

## タスク完了の原則

**このファイルの全タスクが完了するまで作業を継続すること**

### 必須ルール

- 全てのタスクを `[x]` にすること
- 未完了タスク `[ ]` を残したまま作業を終了しない
- 「時間の都合」「難しい」などの理由でのスキップは禁止

### スキップが許可されるケース

技術的理由に該当する場合のみ:

- 実装方針の変更により機能自体が不要になった
- アーキテクチャ変更により別の実装方法に置き換わった

スキップ時は理由を明記:

```markdown
- [~] タスク名 (スキップ理由: 具体的な技術的理由)
```

## 振り返り

### うまくいったこと

- add-feature のワークフロー構造を踏襲し、一貫性のあるコマンドを設計できた
- 「軽微な修正」と「継続作業」の判断基準を定量的に定義できた（1-2ファイル、1時間未満）
- 既存のステアリングに追記する形式により、機能の変更履歴を一箇所で管理できる設計

### 改善点

- implementation-validator から指摘された「実装検証ステップの追加」は今回見送り（次回対応）
- 追加対応セクションに進捗（開始・完了日時）の記載がないため、追跡しにくい可能性

### 次回への学び

- スキル/コマンドの新設時は、既存コマンドとの一貫性を意識してステップ構成を合わせる
- 判断基準は定量的に（ファイル数、時間）定義すると実用的
